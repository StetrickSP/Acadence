# AI Workflow Rules

## Approach

Build this project using a **spec-driven, incremental workflow**.

The context files define:

- what the system does
- how it is architected
- UI conventions
- storage rules
- feature boundaries
- implementation standards

Always implement against these specifications.

Do not invent product behavior, architecture, database structure, or API contracts from scratch.

Favor **small, testable, end-to-end slices** over large speculative implementations.

Every implementation unit should be:

- scoped
- verifiable
- documented
- aligned with architecture.md

---

## Development Philosophy

Follow this order of implementation priority:

1. Data models and schema
2. Backend services
3. API layer
4. Frontend integration
5. Analytics / visualization
6. AI prediction workflows
7. Automation testing

Backend correctness takes priority over UI polish.

Business logic belongs in services — not UI components or API handlers.

---

## Scoping Rules

- Work on **one feature unit at a time**.

- Prefer small, verifiable increments over broad multi-system rewrites.

- Do not combine unrelated boundaries in one implementation step.

- Implement complete vertical slices whenever possible:

  Example:

  **Good scope:**

  Course CRUD

  - SQLAlchemy model
  - service layer
  - API endpoints
  - frontend integration
  - tests

  **Bad scope:**

  Course CRUD + ML prediction + dashboard redesign.

## When to Split Work

Split implementation work if a change combines:

- frontend UI changes + backend analytics changes
- unrelated API routes
- authentication changes + database redesign
- multiple independent feature domains
- undefined behavior not documented in context files
- ML logic changes + unrelated CRUD workflows

If a change cannot be validated quickly end to end, the scope is too broad.

Split it.

---

## Handling Missing Requirements

Do not invent missing behavior.

If a requirement is ambiguous:

1. clarify it in the relevant context file
2. update documentation
3. implement afterward

If requirements are incomplete:

record them as open questions inside:

`progress-tracker.md`

before continuing implementation.

Never silently assume:

- grading weights
- prediction formulas
- ownership rules
- API response formats
- role permissions

---

## Protected Files

Do not modify the following unless explicitly instructed:

- `components/ui/*`
  (generated shadcn/ui components)

- `node_modules/*`

- third-party library internals

- Clerk configuration internals

- autogenerated migration history

- generated ML model artifacts

- lockfiles unless dependency changes are intentional

## Backend Workflow Rules

### API Layer

FastAPI handlers must:

- validate request input
- verify authentication
- enforce authorization
- call services
- return consistent responses

API handlers must NOT:

- contain analytics logic
- contain ML logic
- execute large database workflows directly

Keep handlers thin.

---

### Service Layer

Services own:

- grade calculations
- enrollment logic
- reporting workflows
- analytics orchestration
- prediction orchestration
- business rules

Services must remain framework-agnostic whenever practical.

---

### Database Layer

SQLAlchemy models own:

- schema definition
- relationships
- constraints

Models must NOT contain:

- analytics pipelines
- API logic
- presentation formatting

---

### Analytics Layer

Pandas workflows belong exclusively in:

`backend/analytics/`

Analytics code must be isolated from:

- API request parsing
- UI rendering
- SQLAlchemy schema definitions

---

### ML Layer

ML logic belongs only in:

`backend/ml/`

This includes:

- training
- preprocessing
- prediction
- evaluation
- risk scoring

Prediction systems must remain replaceable.

Do not tightly couple ML models to API handlers.

---

## Frontend Workflow Rules

Frontend responsibilities:

- rendering
- API interaction
- authentication state
- dashboards
- forms
- charts presentation

Frontend must NOT own:

- grading calculations
- analytics computation
- prediction calculations
- authorization decisions

Business truth lives in backend services.

---

## Clerk Auth Workflow

Authentication source of truth:

**Clerk**

Frontend:

- login
- signup
- session state

Backend:

- token validation
- role enforcement
- protected route access

Never trust frontend role claims without backend verification.

## Before Moving to the Next Unit

Before closing a feature unit:

1. Implementation works end to end.

2. No invariant from `architecture.md` was violated.

3. Authentication and authorization remain intact.

4. Tests pass.

5. Documentation reflects the implementation.

6. Build succeeds.

Frontend:

```bash
npm run build
```

Backend:
```bash
pytest
```
